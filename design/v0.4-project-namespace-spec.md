# Enloom v0.4 设计规格 —— 项目级命名空间

日期：2026-06-30
版本：v0.4 spec（待实现）
状态：设计已逐节确认，待 writing-plans / Enloom 自举执行

---

## 一句话定义

v0.4 在 v0.3.3 基础上做一件事：把 `.enloom/` 从「单一全局状态」改为「项目级命名空间」——加一层 `task_board.md` 作唯一入口表，每个项目一个带创建时间戳的目录，内含自己的 `project_state.md` / Registry / tasks / runs / archive。解决「第二次进来路径不一致、跨任务状态全挤在一个文件」的失明问题。

## 问题诊断

v0.3.3 的 `.enloom/` 是单一全局状态模型：

```
.enloom/
  project_state.md   ← 唯一状态真相，所有任务挤在这里
  tasks/T001.md
  runs/T001/...
  archive/task-history.md
```

实际运行的痛点：

- **没有「哪个任务/项目」这一层命名空间**。`T001`、`T002` 是平铺的，新任务接着往同一份 `project_state.md` 塞。
- **跨任务的 Registry、Promise、Open Risks 全混在一个文件**。Compaction 能勉强按 resolved 压缩，但任务边界是隐性的——靠 agent 脑中记住「这次是哪个」，正是要避免的失明。
- **第二次进来路径不一致**：没有稳定锚点定位「该续哪个」，只能靠记忆或扫描。

对比 superpowers 的 per-spec 目录隔离：每个独立工作单元一个隔离目录，第二次进来直接定位。v0.4 把这个隔离机制补进 Enloom。

## brainstorming 决策栈

本次设计由 brainstorming skill 组织，经四项决策逐项确认：

| 维度 | 决定 | 出处 |
|---|---|---|
| 任务粒度 | 顶层目录 = 一个项目，内含多阶段/多任务（非单次运行隔离，而是项目级隔离） | 决策 1 |
| 复用 vs 新建 | 同名项目第二次进来复用已有目录；时间戳=创建时间；项目名去重 | 决策 2 |
| task_board 范围 | 只索引项目（一行一项目），不碰任务/阶段；任务留在各项目 project_state.md | 决策 3 |
| Orient 发现机制 | task_board 驱动 Orient——它是唯一入口表，目录不存在=新项目 | 决策 4 |
| task_board 形态 | 薄表格：project / created / updated / phase / desc | 决策 5 |
| 旧 dogfood 处理 | 冻结当前 `.enloom/project_state.md`（连同旧结构），新任务从零建 | 决策 6 |

## Non-Goals

- 不引入 CLI / scheduler / runtime（守 v0.3 Non-Goals 线）。
- 不改 Registry 七段、Evidence Contract、Ownership/Promise、Compaction Protocol 的内部语义——只改它们住的目录。
- 不做跨项目状态聚合（task_board 只索引，不聚合任务）。
- 不改 trigger 语义（description 不动触发词覆盖，仅在 File Protocol 描述层同步）。

## 第 1 节：核心改动——项目级命名空间

### 目标目录结构

```
.enloom/
  task_board.md                       # 唯一入口表：一行一项目
  2026-06-29-<projectName>/           # 项目目录，时间戳=创建时间，项目名去重
    project_state.md                  # 该项目的状态真相（含 Registry 七段）
    decisions.md
    tasks/
    runs/
    archive/
    prompt-assets/
```

### 五条不变量

1. **task_board 是唯一入口表**。Orient 第一步永远是读它，定位「该续哪个项目」。不在表里 = 新项目。
2. **项目名去重，时间戳固定**。同名项目第二次进来复用已有目录（`<创建时间>-<name>`），只更新 `updated`，不新建。这是解决「第二次路径不一致」痛点的直接机制。
3. **task_board 只索引项目**，不碰任务/阶段。任务留在各项目的 `project_state.md`。避免双重真相。
4. **状态真相下沉到项目目录**。原来全局唯一的 `project_state.md` / Registry / decisions / tasks / runs / archive 全部搬进项目目录。跨项目零污染。
5. **task_board 不删行**。项目关闭后标记 `phase: closed`，行保留（历史可查）。

### task_board.md 形态

```markdown
# Task Board

> 项目级索引。任务详情进各项目 project_state.md。Orient 第一步读本表。

| project | created | updated | phase | desc |
|---|---|---|---|---|
| enloom-dev | 2026-06-29 | 2026-07-02 | v0.3.4 | enloom 产品迭代 |
| art-lab-ingest | 2026-06-30 | 2026-06-30 | ingest | 资料消化 |
```

字段语义：

- `project`：稳定标识，去重键。slug 形态（`enloom-dev`）。
- `created`：创建时间，**永不改**。决定目录名 `YYYY-MM-DD-<project>`。
- `updated`：最近一次进入该项目的日期。每次 Orient/Integrate 后更新。
- `phase`：当前阶段标签（自由文本，如 `v0.3.4` / `ingest` / `closed`）。
- `desc`：一句话简介。

### Orient 流程改写

```
1. 读 .enloom/task_board.md
2. 定位目标项目行（用户指明 / 唯一活跃项目 / 最近 updated）
3. cd .enloom/<created>-<project>/
4. 读该目录的 project_state.md（含 Registry 七段）→ 续上一个阶段
5. 若 task_board 不存在或无匹配行 → 当新项目：
   a. 创建 .enloom/<今天>-<projectName>/ 目录
   b. 初始化该项目的 project_state.md
   c. 在 task_board.md 加一行（created=今天）
```

## 第 2 节：改动清单

按「契约层 → 引用层 → 产物层」三组。逻辑层（Registry/Evidence/Ownership/Compaction 内部语义）不动，只改它们引用的路径。

| 文件 | 改动 |

### C. 产物层（dogfood 自举）

| 文件 | 改动 |
|---|---|
| `.enloom/project_state.md`（当前 dogfood） | 冻结，连同旧结构作 v0.3.3 单状态模型的历史证据；新结构留给实现本改动的任务从干净状态建立 |

### 迁移策略

当前 `.enloom/project_state.md` 是 v0.3.3 rename 的 dogfood 记录。**不就地迁移，冻结留证**（类似 `AgentOS/` 快照的处理逻辑，理由一致：强行改写会篡改历史证据，且 dogfood 内容是已完成的旧任务、续不上）。新结构用「实现 v0.4 这个任务本身」从干净状态建立，成为新设计的第一个 dogfood 样本。

## 第 3 节：验证标准

实现完成的判据：

1. **结构正确**：新任务进入 Enloom 时，会建 `.enloom/<日期>-<项目>/` 目录 + 在 task_board 加行，而非往全局 `project_state.md` 塞。
2. **路径引用一致**：grep 全 skill 包，无残留的「裸 `.enloom/project_state.md`」（应均为 `.enloom/<project>/project_state.md` 或带前缀）。
3. **复用机制成立**：同名项目第二次进入，识别已有目录并复用，只更新 `updated`，不新建目录。
4. **Orient 可定位**：Orient 第一步读 task_board，能正确路由到目标项目目录。
5. **逻辑层零回归**：Registry/Evidence/Ownership/Promise/Compaction 内部语义不变，只是换了住址。
6. **自举落地**：实现本改动的 dogfood 记录本身落在新结构下（`.enloom/<日期>-enloom-v04/`），作为第一个干净样本。

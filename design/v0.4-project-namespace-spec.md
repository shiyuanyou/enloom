# Enloom v0.4 设计规格 —— 项目级命名空间 + 落盘时序契约

日期：2026-06-30
版本：v0.4 spec（待实现）
状态：设计已逐节确认，待 writing-plans / Enloom 自举执行

> **修订记录**：本 spec 初版只含「项目级命名空间」。art_lab 实跑诊断后，增补「落盘时序契约」——两者正交但同属 v0.4。

---

## 一句话定义

v0.4 修 enloom 实跑暴露的两块缺口：

1. **项目级命名空间** —— `.enloom/` 从单一全局状态改为「task_board 入口表 + 每项目一目录」，解决第二次进来路径不一致、跨任务状态全挤一个文件。
2. **落盘时序契约** —— 给生命周期每个 Stage 加入口/出口闸门（文件存在性）+ 主流程↔worker 握手时序图，解决 task packet / worker report / 中间状态根本不落盘（art_lab `tasks/` `runs/` 全空）。

---

## 问题诊断

### 缺口一：命名空间（初版已述）

v0.3.3 的 `.enloom/` 是单一全局状态模型，所有任务挤在一个 `project_state.md`，跨任务的 Registry / Promise / Open Risks 全混，第二次进来无稳定锚点定位「该续哪个」。

### 缺口二：落盘（art_lab 实跑暴露）

art_lab 的 `.enloom/` 跑过 3 个 Phase、动用 8 worker + control agent、~90 分钟，`project_state.md` 写满成果——但 **`tasks/` `runs/` `archive/` 全空**。90 分钟工作的中间决策、worker 时序、为什么这样分派，全随会话消失。

**根因三重（不是 bug，是系统性强制力真空）：**

| 根因 | 表现 |
|---|---|
| **时序未命令化** | task-packet / worker-report 模板字段完整，但 workflow-steps 全是动词短语（「make packet / dispatch」），从未说「写进 `runs/<TASK>/task.md` 这个文件」。agent 可全程在上下文里「走完」六阶段，一次磁盘不碰。 |
| **时序流程未落盘** | SKILL.md File Protocol 只画静态「住址」（什么文件住哪），不回答「谁、何时、怎么写进那个住址」。主流程↔worker 握手从未成协议。 |
| **铁律 2/5 无机械闸门** | 铁律 4 被 mechanize 成 Evidence Contract 硬约束（PASS iff 全 check 跑了 + evidence 非空）；但铁律 2「无任务包不派 worker」、铁律 5「无状态更新不归档」没有「dispatch 前 health-check 证明 task.md 存在」的机械检查。 |

共同点：enloom 把**文件协议**（住址）和**执行纪律**（谁、何时、怎么写进住址）混在一起，只认真写了前者。缺一层「Stage X 必须执行命令 Y、产出文件 Z 落盘」的时序契约。

## brainstorming 决策栈

本次设计由 brainstorming skill 组织，经多轮决策确认：

| 维度 | 决定 | 出处 |
|---|---|---|
| 命名空间：任务粒度 | 顶层目录 = 一个项目，内含多阶段/多任务 | 决策 1 |
| 命名空间：复用 vs 新建 | 同名项目复用已有目录；时间戳=创建时间；项目名去重 | 决策 2 |
| 命名空间：task_board 范围 | 只索引项目，不碰任务 | 决策 3 |
| 命名空间：Orient 发现机制 | task_board 驱动 Orient——唯一入口表 | 决策 4 |
| 命名空间：task_board 形态 | 薄表格：project/created/updated/phase/desc | 决策 5 |
| 命名空间：旧 dogfood 处理 | 冻结当前 .enloom/project_state.md，新任务从零建 | 决策 6 |
| 落盘：纳入 v0.4 还是拆 v0.5 | **纳入 v0.4，与命名空间同修** | 决策 7 |
| 落盘：时序契约形式 | **闸门表 + 时序图都要** | 决策 8 |
| 落盘：闸门谁执行 | **双保险——control 自检 + health-check 硬闸门** | 决策 9 |

## Non-Goals

- 不引入 CLI / scheduler / runtime（守 v0.3 Non-Goals 线）。
- 不改 Registry 七段 / Evidence Contract / Ownership-Promise / Compaction Protocol 的内部语义。
- 不做跨项目状态聚合（task_board 只索引项目，不聚合任务）。
- 不动 description 触发词覆盖。
- 不引入真正的多进程隔离（仍是单/多 agent 在文件协议上协作；worker 间真实隔离的「blind spot」继续在 report 显式声明）。

## 第 1 节：项目级命名空间（初版内容，保留）

### 目标目录结构

```
.enloom/
  task_board.md                       # 唯一入口表：一行一项目
  2026-06-29-<projectName>/           # 项目目录，时间戳=创建时间，项目名去重
    project_state.md                  # 该项目的状态真相（含 Registry 七段）
    decisions.md
    tasks/
    runs/
    archive/
    prompt-assets/
```

### 五条不变量

1. **task_board 是唯一入口表**。Orient 第一步永远是读它，定位「该续哪个项目」。不在表里 = 新项目。
2. **项目名去重，时间戳固定**。同名项目第二次进来复用已有目录（`<创建时间>-<name>`），只更新 `updated`，不新建。
3. **task_board 只索引项目**，不碰任务/阶段。任务留在各项目的 `project_state.md`。
4. **状态真相下沉到项目目录**。跨项目零污染。
5. **task_board 不删行**。项目关闭后标记 `phase: closed`，行保留。

### task_board.md 形态

```markdown
# Task Board

> 项目级索引。任务详情进各项目 project_state.md。Orient 第一步读本表。

| project | created | updated | phase | desc |
|---|---|---|---|---|
| enloom-dev | 2026-06-29 | 2026-07-02 | v0.3.4 | enloom 产品迭代 |
```

字段：`project`（去重键，slug）/ `created`（永不变，定目录名）/ `updated`（每次进入更新）/ `phase`（当前阶段标签，closed 表关闭）/ `desc`（一句话简介）。

### Orient 流程改写

```
1. 读 .enloom/task_board.md
2. 定位目标项目行（用户指明 / 唯一活跃 / 最近 updated）
3. cd .enloom/<created>-<project>/
4. 读该目录 project_state.md（含 Registry 七段）→ 续上一阶段
5. task_board 不存在或无匹配 → 当新项目：建目录 + 初始化 project_state + task_board 加行
```

## 第 2 节：落盘时序契约（新增，art_lab 诊断后）

### 2.1 闸门表 —— 每个 Stage 的入口/出口机械检查

闸门 = **文件存在性检查**，机械、可验证。control agent 在 Stage 入口自检 + health-check 作硬闸门，双保险。

| Stage | 入口闸门（进本阶段前必须满足） | 出口闸门（离开本阶段前必须满足） |
|---|---|---|
| 0 Triage | — | triage 结果已定（direct/light-plan 不进闸门表，只有 enloom 走下面） |
| 1 Orient | task_board.md 存在（首次则创建） | 5-10 行状态摘要已写入 project_state 的 Current Phase（或确认无需） |
| 2 Plan | 已定位项目目录（<created>-<project> 存在） | `tasks/phase-plan-<phase>.md` 存在；phase goal 清晰（gate check 通过） |
| 3 Execute（per task） | **`runs/<TASK>/task.md` 存在** ← 铁律 2闸门 | **`runs/<TASK>/output.md` 存在** + **`runs/<TASK>/report.md` 存在** + report 有 Result 段（done/blocked/failed） |
| 4 Verify | `runs/<TASK>/report.md` 存在（含 Evidence Contract 四要素） | `runs/<TASK>/report.md` 的 Review Result 段已填（verdict + conclusion） |
| 5 Integrate | 所有本阶段 task 的 report Review Result 段已填 | project_state.md + Registry 已更新；compaction 触发检查已跑 |
| 6 Close | Integrate 出口闸门全过 | `archive/<phase>-entry.md` 存在；旧 runs/ 内容已归档或声明丢弃 |

**执行规则（双保险，决策 9）：**

- **control 自检**：control agent 在每个 Stage 入口跑 `ls runs/<TASK>/task.md` 之类的存在性检查。缺则退回上一 Stage 补齐，不准越过。
- **health-check 硬闸门**：Stage 转移时必须跑 health-check，它检查上一阶段的出口闸门文件，不达标报 drift finding（铁律违反信号）。control 必须先补齐才能进下一 Stage。

### 2.2 时序握手图 —— 主流程 ↔ worker 落盘时序

`<TASK>` 的完整生命周期，按磁盘操作排列。每一步箭头都是一次落盘，不是上下文传递。

```
control agent                worker                disk (.enloom/<project>/)
    |                           |                          |
    |--[Stage 2 Plan] 写 tasks/phase-plan-<phase>.md------->|
    |                           |                          |
    |--[Stage 3 入口闸门] 检查 runs/<TASK>/task.md -------->|
    |   不存在 → 退回 Plan; 存在 → 继续                      |
    |                           |                          |
    |--- make-prompt: 写 runs/<TASK>/task.md ------------->|  ← 铁律 2 落地
    |                           |                          |
    |--- dispatch: 把 task.md 路径交给 worker ------------->|
    |                           |--- 读 task.md ---------->|
    |                           |--- 执行(读 Inputs) ----->|
    |                           |--- 写 runs/<TASK>/output.md -->|  ← 产出落地
    |                           |--- 写 runs/<TASK>/report.md --->|  ← 报告落地
    |                           |    (含 Result/Evidence/Not Checked/Blind Spots)
    |<-- done/blocked/failed ----|                          |
    |                           |                          |
    |--[Stage 3 出口闸门] 检查 output.md + report.md ----->|
    |   + report 有 Result 段 → 通过; 缺 → 退回 worker       |
    |                           |                          |
    |--[Stage 4 review] 读 report.md --------------------->|
    |--- 写 report.md 的 Review Result 段(verdict+concl) ->|  ← review 落地
    |                           |                          |
    |--[Stage 5] 更新 project_state.md + Registry -------->|
    |                           |                          |
    |--[Stage 6] 写 archive/<phase>-entry.md ------------->|
    |--- 归档/声明丢弃 runs/<TASK>/ 内容 ----------------->|
```

**关键：worker 的产出和报告必须落盘成文件，不能只留在对话上下文。** dispatch 不是「把任务说给 worker 听」，是「把 task.md 路径交给 worker，worker 写 output.md/report.md」。这是闸门表能成立的物理前提。

### 2.3 铁律 2/5 的机械化升级

| 铁律 | v0.3（陈述） | v0.4（机械化） |
|---|---|---|
| **铁律 2** 无任务包不派 worker | 陈述 | **dispatch 前闸门：`runs/<TASK>/task.md` 必须存在**。control 自检 + health-check 双查。不存在 → 不准 dispatch，退回 Plan。 |
| **铁律 5** 无状态更新不归档 | 陈述（扩展：registry 处理） | **archive 前闸门：所有 task 的 `report.md` 的 Review Result 段已填 + project_state/Registry 已更新**。health-check 硬查。缺 → 不准 archive。 |

铁律 4（Evidence Contract）已机械化，v0.4 不动；2/5 对齐 4 的机械化标准。

## 第 4 节：验证标准

1. **命名空间成立**：新任务建 `.enloom/<日期>-<项目>/` + task_board 行，而非全局单状态。
2. **路径引用一致**：grep 全包无残留裸 `.enloom/project_state.md`。
3. **复用机制**：同名项目第二次进入复用目录，只更新 updated。
4. **逻辑层零回归**：Registry/Evidence/Ownership/Promise/Compaction 语义不变。
5. **闸门可执行**：每个 Stage 的入口/出口闸门都是「文件存在性」检查，control 自检 + health-check 双跑。
6. **落盘时序成文**：landing-contract.md 含完整闸门表 + 握手时序图，workflow-steps 每 Stage 引用它。
7. **铁律 2/5 机械化**：dispatch 前必有 task.md，archive 前必有 Review Result 段，均有 health-check 硬查。
8. **自举落地**：实现本改动的 dogfood 落在 `.enloom/<日期>-enloom-v04/`，且**自身严格遵守闸门表**（每个 task 的 task.md/output.md/report.md 真实存在）——这是验证标准 #5/#7 的活样本。

## art_lab 经验的二分（设计的理论依据）

art_lab 原始样本（POSTMORTEM / PARALLEL_BATCH_WORKFLOW / AUDIT_TEMPLATE / SUBAGENT_TEMPLATE）经提炼，可干净拆成两层：

| 层 | art_lab 原始形态 | v0.3 通用结构 |
|---|---|---|
| **协议层**（约束 agent 行为，靠读文档就能遵守） | 命名规范、防重复搜索、partial 拒绝、registry 登记、抽查、白名单 | Registry / Ownership Table / Promise Registry / Compaction |
| **证据层**（验证 agent 结果，必须留检查痕迹） | 死链 one-liner、quote 编码扫描、引用计数、点检 | Evidence Contract + Audit Packet 的命令证据要求 |

分界判据：凡是「人/agent 凭读文档就能避免」的 → 协议层；凡是「必须实际跑命令验证文件系统状态才能发现」的 → 可执行证据。前者是「约束 agent 行为」，后者是「验证 agent 结果」——这正是 task packet（约束）与 audit packet（验证）的分野。

## 第 1 节：生命周期主架

### 核心改动

SKILL.md 从「6 操作菜单」重写为「生命周期阶段驱动」。操作不消失，降级为阶段内的子动作。

### 生命周期六阶段（融合现有 Step 0-7 与 art_lab P0-P4，去 wiki 化）

```
0. Triage（入口判断）   → 决定 direct / light-plan / enter lifecycle
1. Orient（恢复状态）   → 读 project_state + registry + 最近 report
2. Plan（规划当前阶段） → phase goal + ownership table + promise registry 草案
3. Execute（派 worker） → make task packet / dispatch / worker 在边界内发挥
4. Verify（验收证据）   → review report + audit packet + evidence gate
5. Integrate（压缩进状态）→ 更新 project_state + registry + compaction 检查
6. Close（归档退出）    → archive + 闭合校验 + 用户汇报
```

### 与现有 Step 0-7 的映射

| 现有 Step | v0.3 生命周期阶段 | 说明 |
|---|---|---|
| Step 0 Triage | 阶段 0 Triage | 不变 |
| Step 1 Orient | 阶段 1 Orient | 不变 |
| Step 2 Plan | 阶段 2 Plan | 不变 |
| Step 3 Make Packet + Step 4 Dispatch | 阶段 3 Execute（**合并**） | packet 制作与 dispatch 连续，分开是人为割裂 |
| Step 5 Review | 阶段 4 Verify | 重命名，强调「验收证据」 |
| Step 6 Integrate | 阶段 5 Integrate | 不变 |
| Step 7 Archive | 阶段 6 Close | 重命名 |

### 操作降级为子动作

现有 6 操作映射为生命周期阶段内的子动作：

| 子动作 | 所属阶段 |
|---|---|
| `triage` | 阶段 0 |
| `plan` | 阶段 2 |
| `make-prompt` | 阶段 3 Execute 内 |
| `dispatch` | 阶段 3 Execute 内（原隐含在 dispatch 步骤，现在显式化） |
| `review` | 阶段 4 Verify 内 |
| `audit` | 阶段 4 Verify 内（**新增**，周期性/发布前） |
| `archive` | 阶段 6 Close |
| `health-check` | 阶段 1 Orient（漂移检查）+ 阶段 4 Verify（周期性） |

health-check 从独立操作降为 Orient/Verify 的周期性触发项，不再是顶层操作。

### 五条铁律（保持不变）

铁律是生命周期的约束，不是操作的约束，v0.3 全部保留：

1. No AgentOS without trigger
2. No Worker without Task Packet
3. No Parallel without Ownership（v0.3 升级：No Parallel without Ownership **Table**）
4. No PASS without Evidence（v0.3 机械化：见 Evidence Contract 硬约束）
5. No Archive without State Update（v0.3 扩展：含 registry 风险区段已处理）

## 第 2 节：Registry 多区段

### 问题陈述

art_lab 核心教训：state 只记「完成了什么」会留系统性漏洞，真正能恢复项目的是「未闭合风险清单」。

### Registry 七区段（project_state.md 的固定结构组成部分，不拆独立文件）

| 区段 | 通用含义 | art_lab 原型 |
|---|---|---|
| `## Active Tasks` | 进行中的任务（含状态） | state.md 的 batch 状态 |
| `## Promised Outputs` | 已声明但未兑现的产出（谁承诺、谁依赖、校验点） | Promise Pages 承诺清单 |
| `## Pending Dependencies` | 跨任务/跨阶段的未满足依赖 | 待激活链接 |
| `## Broken References` | 已发现的断裂引用（源→目标、状态、备注） | Dead Link Registry |
| `## Known Exceptions` | 有意保留的例外（白名单，不计为缺陷） | `[[index]]/[[log]]` 导航伪死链 |
| `## Accepted With Risk` | 已验收但带风险、需复查的项 | accepted-with-risk 结论 |
| `## Rejected Reports` | 被拒报告索引（失败信号，防重试相同路径） | —（art_lab 缺口，v0.3 补） |

### 通用化关键

art_lab 的 dead-link、promise page 是 wiki 领域词。通用化后：
- `Broken References` 涵盖任何「产物里声明了指向，但目标不存在/断裂」——死链、失效 import、不存在的函数引用、悬空 TODO 指向。
- `Promised Outputs` 涵盖任何「worker A 声明将产出 X，worker B 已引用 X」的前向声明。

`## Rejected Reports` 是 v0.3 对 art_lab 缺口的补充：失败报告留索引，防止重试相同失败路径。

### Registry 更新纪律

- **Verify 阶段（阶段 4）**：review 后，control agent **必须**把发现的问题登记进对应区段（broken reference、accepted-with-risk、rejected）。
- **Integrate 阶段（阶段 5）**：registry 更新是 archive 的前置（铁律 5 的扩展）。
- **Orient 阶段（阶段 1）**：control agent 读 project_state 时**必须扫 registry 的风险区段**（Pending Dependencies / Broken References / Accepted With Risk / Rejected Reports）。这是「主 agent 可薄但不能盲」的落点——薄编排器的价值就在于守住全局不变量。

### 与 Compaction 的关系

Registry 是 project_state 的活性部分，Compaction 时 registry 区段优先保留，只有当区段项被标记 resolved/closed 时才移入 archive。保证「未闭合风险清单」永远是当前真相。

### 放在哪

- `templates/project-state.md` 扩展，七区段写成固定骨架（填空式）。
- Registry 更新纪律并入 `references/registry-and-compaction.md`（见第 5 节结构预览）。

## 第 3 节：Evidence Contract + Audit Packet

art_lab 最硬的一块——「6 次审计全 PASS 仍 16+ 死链」的根因。

### 3a. Evidence Contract（证据契约）

**四要素**（任何宣称完成的 worker 必须产出）：

1. **Checks Run** —— 执行了哪些验证（具名）
2. **Evidence** —— 每项验证的证据（命令输出 / 文件路径 / 引用）
3. **Not Checked** —— 哪些该验证但没验证（显式声明盲区）
4. **Known Blind Spots** —— 这些盲区为什么没查、风险多大

**硬约束（补 art_lab AUDIT_TEMPLATE 最大缺陷）**：

> `verdict = PASS` **当且仅当**所有 declared checks 已执行且 evidence 非空。
> 任何 declared check 标记 `NOT RUN` → verdict 不能是 PASS。
> 任何 high-severity issue 未解释 → verdict 不能是 PASS。
> PASS without evidence → 自动降级为 `needs-rework`（铁律 4 的机械化）。

art_lab 的 AUDIT_TEMPLATE 只靠结构间接防空泛（每检查项绑定命令但没显式禁止无证据 PASS），v0.3 必须显式写死这条硬约束。

**三态 verdict**（保留 art_lab 设计，别退化成二值）：

- `PASS` —— 所有必检通过、证据齐、盲区已声明
- `ISSUES` —— 有缺陷但可继续（medium/low severity，已登记进 registry）
- `FAIL` —— high severity 未解 / 必检未跑 / 证据缺失

ISSUES 中间态支撑 control agent「带已知缺陷继续推进」的决策，避免二值 verdict 逼着 agent 要么全过要么全停。

### 3b. Audit Task Packet（审计任务包）

**check_item 标准 5 元组 schema**（art_lab 隐含此结构，v0.3 显式 schema 化）：

```
check_item:
  id:              唯一标识（如 dead-link-scan）
  command:         可执行检查命令（或 "manual: 描述" for 人工检查）
  pass_condition:  通过判据（如 "输出为空"）
  fail_signal:     失败信号（如 "出现任何 DEAD: 行"）
  named_list:      产出具名清单名（如 dead_links，用于登记进 registry）
```

**Audit Packet 模板**（task-packet.md 的特化）：

```
# Audit Task Packet: Axxx

audit_mode:   batch（抽样）/ final（全量）
audit_scope:  范围描述（如"当前阶段所有产物"）

## Goal
## Anti Goal
## Inputs

## Checks
  - id: ... | command: ... | pass: ... | fail: ... | named_list: ...

## Conclusion Rules
  - 任何 NOT RUN critical check => FAIL
  - 任何 high severity => FAIL
  - PASS 必须附 command evidence + blind spots

## Severity Levels
  high（阻断）/ medium（可延后）/ low（登记即可）

## Output Lists（具名清单，用于登记 registry）
  dead_links / broken_refs / malformed_outputs / ...

## Return To Caller（裁剪规则）
  主 agent 只接收 verdict + 计数，详情下沉 registry

## Done Signal
  PASS | ISSUES | FAIL
```

**Audit Packet vs 普通 Task Packet**：

- 普通 packet：约束 worker **行为**（goal/anti-goal/writable/forbidden）——「该做什么」
- Audit packet：约束 worker **验证**（checks/evidence/conclusion rules）——「该验证什么、怎么算通过」

两者正交。一个 worker 可同时收到行为约束 + 验证要求（普通 packet 的 Required Verification 字段引用 audit check_item），也可单独 dispatch 一个纯审计 worker。

### 生命周期挂点

- **Execute 阶段（阶段 3）**：普通 task packet 的 `Required Verification` 字段可内联 check_item，或指向一个 audit packet。
- **Verify 阶段（阶段 4）**：`audit` 子动作——周期性（每 N 个 worker）或发布前 dispatch audit packet。art_lab 的 batch_audit（每 5 批）/ final_lint（全量）映射为这里的 batch / final 两种 audit_mode。

### 领域隔离

art_lab 的具体命令（`rg '\[\[...\]\]'` 死链扫描、`find ... -name '*"*'` quote 编码扫描）是 wiki 领域专属，**不进通用模板**。通用 audit packet 只提供 5 元组 schema（插槽），具体 command 由当前任务的 domain 注入。art_lab 的这些命令作为 worked example 写进 `references/examples/art-lab-worked-example.md`。

### 放在哪

- 新增 `references/evidence-contract.md` —— 四要素 + 硬约束 + 三态 verdict。
- 新增 `templates/audit-task-packet.md` —— 5 元组 check_item 填空模板。
- 扩 `templates/worker-report.md` —— 对齐 Evidence Contract 四要素（现有模板已有 Verification 和 Known Blind Spots，需补 `Not Checked` 显式声明）。

## 第 4 节：Ownership Table + Promise Registry 双表

并行安全的物理基础。art_lab 证明：并行不是策略，而是所有权明确后的例外优化。

### 4a. Ownership Table（文件所有权表）

**三阶所有权模型**（任何并行 agent 工作流适用，本质是「全局唯一可变状态无法并行」）：

| 阶 | 通用语义 | 谁能写 | art_lab 原型 |
|---|---|---|---|
| **并行写区** | 各 worker 独占的新产物 | 对应 worker 独占，互不相交 | P2 各 lane 的新建页 |
| **串行集成区** | 全局唯一可变状态 | 单线程 worker 独占 | P3 的 index.md/log.md/回填 |
| **只读区** | 不可变输入 | 永远只读 | raw/ |

**双清单纪律**（art_lab 血泪教训）：给 worker 的 packet 必须**同时**提供：
- `Writable Files` —— 独占清单（精确到路径）
- `Forbidden Files` —— 禁碰清单（**显式列出全局状态文件**，不能只说「别碰共享文件」）

art_lab 踩的坑是「worker 顺手改 index/log 破坏所有权」。正解是 packet 里**显式禁碰**。当前 task-packet.md 模板已有 Writable/Forbidden 两字段，v0.3 强化：**并行 dispatch 时，Forbidden 必须显式枚举串行集成区的文件**。

**所有权表模板**（并入 plan 阶段的 phase-plan）：

```
## Ownership Table（并行 dispatch 时必填）

| 资源/路径 | 阶 | 可写者 | 阶段 |
|-----------|-----|--------|------|
| 新产物 A  | 并行写区 | Worker-1 独占 | Execute |
| 新产物 B  | 并行写区 | Worker-2 独占 | Execute |
| project_state / registry | 串行集成区 | Control agent 独占 | Integrate |
| raw 输入  | 只读区 | 无人 | 全程 |
```

**选型约束**（art_lab 隐含，v0.3 显式写明）：串行集成区存在的前提是「全局状态无法 append-only 并行」。如果某任务的全局状态可以设计成 append-only log（无冲突合并），理论上集成区也能并行——但 v0.3 保守地**默认串行集成区强制单线程**，除非 plan 阶段显式论证可并行。守住「并行是例外」的铁律。

**生命周期挂点**：
- **Plan 阶段（阶段 2）**：选择 parallel/hybrid 策略时，所有权表是**硬前置**（铁律 3 升级：No Parallel without Ownership Table）。
- **Execute 阶段（阶段 3）**：make-prompt 生成 packet 时，从所有权表派生每个 worker 的 Writable/Forbidden 双清单。

### 4b. Promise Registry（承诺产出注册表）

art_lab 的 Promise Pages = 「前向声明产出 + 悬空引用容忍 + 末端校验」。通用化为并行协议。

**Promise Registry 字段**（占 Registry 七区段的 `## Promised Outputs`）：

```
promised_output:
  declarer:    声明者（哪个 worker/任务承诺创建）
  identifier:  目标标识符（产物名/路径/符号）
  consumers:   依赖者（哪些 worker 已引用它）
  verify_at:   校验点（通常 Verify 阶段）
  status:      promised | fulfilled | broken
```

**运作周期**：

1. **Plan 阶段**：control agent 起草 promise registry 草案（哪些产出将被前向声明、谁依赖）。
2. **Execute 阶段**：worker A 可引用 worker B 将创建的产出（写悬空引用），不阻塞；每次引用登记进 `## Promised Outputs`。
3. **Verify 阶段**：audit check 校验所有 promised outputs 是否 fulfilled。未兑现 = broken，登记进 `## Broken References`，触发点对点修复或降级。

**降级机制**（art_lab 原本是「必须修」，v0.3 补显式降级路径，因不是所有存储都容忍悬空引用）：

- 存储容忍悬空引用（如 wiki/文件系统的 dangling link）→ 引用先写，末端校验兑现
- 存储不容忍（如代码 import、强类型符号）→ 不能用 promise 机制，必须串行（worker B 先完成，worker A 才能引用）

**选型判据写进 references**：control agent 在 Plan 阶段判断「本任务的产物引用层是否容忍 dangling reference」——容忍才允许 promise 机制 + 并行，不容忍则相关任务强制串行。这把 art_lab 隐含的 Obsidian 约束显式化为通用决策点。

### 放在哪

- Ownership Table + Promise Registry 完整纪律并入 `references/registry-and-compaction.md`（与 Registry、Compaction 共享一份「状态治理」reference，避免文件碎片）。
- Ownership Table 骨架并入 `templates/phase-plan.md`。
- Promise Registry 字段格式并入 `templates/project-state.md` 的 `## Promised Outputs` 区段说明。

## 第 5 节：Compaction Protocol

### 问题陈述

art_lab 隐性教训：状态文档会膨胀，「压缩进状态」如果不带压缩纪律，就只是把聊天上下文膨胀转移到文档膨胀。当前 skill 有 compaction 的概念（<200 行触发），但没有协议——什么时候触发、压缩什么、压到哪、怎么保证不丢未闭合风险。

### 核心原则

**压缩的是「已闭合的过程细节」，绝不压缩「未闭合风险」。** 这是与 Registry 的直接接口：Registry 七区段是活性真相，compaction 时优先保留；只有当某项被标记 resolved/closed 时才移出。

### Compaction Protocol 四步

**触发条件**（任一满足）：

- project_state.md 超过 ~200 行
- project_state 的 `## Accepted Results` 区段（v0.2 已有此区段）累积超过阈值（如 10 条已归档结果）——注意这是 project_state 顶层区段，不是 Registry 七区段之一
- 新会话 Orient 阶段读 project_state 超过 3 分钟读不完（主观信号，但重要）

**四步流程**：

| 步 | 动作 | 压缩对象 | 去向 | 保留 |
|---|---|---|---|---|
| 1. 扫描 | 遍历 project_state 各区段 | 已标记 resolved/closed 的 registry 项 + 已归档阶段的 Accepted Results | — | — |
| 2. 迁移 | 把步骤 1 识别的项移入 `archive/` | 过程细节、已完成任务的中间状态 | `archive/phase-history.md` | 仅留一句结论在 state |
| 3. 收口 | 每个被压缩的阶段留一行索引 | — | project_state 加 `## Archived Phases` 索引 | "Phase X 完成于 Y，结论 Z，详见 archive/" |
| 4. 校验 | compaction 后跑 health-check | — | — | registry 风险区段（Pending/Broken/Accepted-with-Risk/Rejected）**条目数不减少** |

**第 4 步是防错关键**：compaction 最危险的失败是「压缩时误删了未闭合风险」。校验规则——**compaction 前后，Registry 的四个风险区段（Pending Dependencies / Broken References / Accepted With Risk / Rejected Reports）的条目数只能持平或减少（仅当某项 genuinely resolved），绝不能因压缩而消失**。如果条目数意外减少，说明压缩误删，必须回滚。

### Compaction 与生命周期的关系

- **Integrate 阶段（阶段 5）**：每次 integrate 后**检查触发条件**。满足则在本阶段内执行 compaction（不必单独成阶段）。
- **Orient 阶段（阶段 1）**：新会话恢复时，如果发现 project_state 已超阈值但未 compact，记一条 health-check 发现（漂移信号）。
- **Close 阶段（阶段 6）**：archive 之前可选触发一次 compaction，保证退出时 state 干净。

### 放在哪

- Compaction Protocol 写进 `references/registry-and-compaction.md`（与 Registry、Ownership、Promise 共享「状态治理」reference）。

### 6c. 自举验证策略

按决策 5「全程用 AgentOS 自身跑这次开发」。用 v0.3 自己新增的纪律来管 v0.3 的实现，以此验证纪律好不好用。

**自举闭环**：

1. **现在**：本 spec 写进 `AgentOS/project_state.md`（作为 plan 阶段的输入）。
2. **实现阶段**：进 AgentOS 生命周期——Orient（读 spec）→ Plan（拆 v0.3 的实现任务，填 Ownership Table）→ Execute（make task packet 改各文件，用 Evidence Contract 要求每个改动附验证）→ Verify（audit：跑 validation.md 的结构校验 + 证据检查）→ Integrate（更新 project_state + registry）→ Close（archive + compaction）。
3. **验证点**：实现过程中，v0.3 新增的几条硬约束会被真实触发：
   - 改 SKILL.md 时，task packet 的 Forbidden 要显式枚举（验证双清单纪律）

   - 多个 reference 可以并行改（Ownership Table 决定谁改哪个文件）

4. **诚实约束**（沿用 v0.2 decisions.md D5）：单 agent 环境下，worker 隔离靠**字段纪律**（packet 的 Writable/Forbidden）而非**执行隔离**（没有独立 subagent runtime）。这是已记录的限制，v0.3 不假装解决它，但在 Evidence Contract 里要求诚实声明这个盲区（Not Checked: 跨 worker 真实隔离性）。

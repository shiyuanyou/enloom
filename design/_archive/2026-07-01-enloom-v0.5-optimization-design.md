## 0.A. 评审 meta-observation（理解全部裁决的前提）

本轮诊断依据是 **dev-wiki 知识库交叉评估**，而该 wiki 沉淀自 **art_lab 31 批 ingest 流水线**——一个真实的、大批量的、**多 sub-agent 真实并行**的工程场景。但 enloom 和 clear-mind **当前都在单 agent 串行模式下运行**：

- enloom `scheduler-rules.md:53` 自己承认："在单个 control agent 会话里，无法真正并行 dispatch 多个写 worker……执行实际串行。"
- enloom `glossary.md:12`："worker 可以是 sub-agent、Pi、或当前 agent 进入 worker mode"。

这造成一个**系统性偏差**：初稿把"大规模并行流水线的教训"移植到了"尚未达到那个规模的 skill"上。因此：

- 凡是**针对真实并行编排**的建议（S1 计数、S3 分组），价值被高估，应降级或推迟；
- 凡是**针对文档完整性 / 闸门机械化 / 诚实盲区**的建议（S5、S4、X2），与规模无关，是真正的结构性补课，应优先。

本 meta-observation 同时适用于 clear-mind v0.3 评审。

## 3. 7 条局限与优化建议（含评审裁决）

### S1 · Evidence Contract 未覆盖 sub-agent 自报不可信类失败

**评审裁决：🟡 吸取机制，但重定性为「report-vs-output 一致性校验」，优先级 高→中**

**现状**：Evidence Contract 四要素（Checks Run / Evidence / Not Checked / Known Blind Spots）覆盖"check 是否跑了 + 证据是否非空"，但漏洞是——sub-agent 可以声称"处理了 107 个条目"但实际产出是 130 个（低估 -23）或高报 +17。数量偏差方向不固定，不可校准。证据契约对此无感——因为"107 个条目"作为 claim 被写在 evidence 段里，gate 无法判定它不属实。

**wiki 依据**：[[Sub-agent计数不可信]]（方向不固定的实证：建筑 A1+A2 低估 -23、电影 A1 高报 +3、A2 高报 +17、雕塑 A3 低估 -1）。[[闸门前置]] 闸门清单 #1 即"条目计数自洽——主控独立计数，不信任 sub-agent 自报"。

**原建议**：Evidence Contract 加第 5 项检查维度：**Independent Verification**（独立验证）。对每一类 sub-agent 可公开报表的 claim（条目计数、通过率、覆盖率等），Verify 阶段要求主控用独立脚本（`grep -c`、`awk` 逐行计数）比对 sub-agent 自报。偏差 > 0 → 自动标记为 ISSUES（非 PASS），登记到 Registry `## Broken References`。

**改动范围**：`evidence-contract.md`（加 §Independent Verification）+ `review-checklist.md`（加计数自洽检查）+ `templates/task-packet.md`（Required Verification 段加 "Countable outputs: [list]" 字段）。

> **【评审裁决】** 原建议套用 `[[Sub-agent计数不可信]]` 框架，而该 wiki 页的实证（主控 vs sub-agent 计数偏差）**预设了真正独立的 sub-agent 运行时**。当前主流是单 agent 串行（`scheduler-rules.md:53` 自承认；`glossary.md:12` worker 定义含"当前 agent 进入 worker mode"），"sub-agent 自报计数"基本退化成"agent 对自己工作量的自报"——此时"独立验证"的对手方不再是独立 runtime，而是 agent 自己刚写的 output 文件。
>
> 但机制本身（Verify 阶段用 `grep -c` / `awk` 独立比对 worker-report 的 claim 数与 output 实际条数）是**合理且廉价的完整性检查**，无论 worker 是谁。
>
> **裁决**：
> 1. 加 Evidence Contract 第 5 维，但**定性为 "report-vs-output claim 一致性校验"**，不叫 "sub-agent 计数验证"——后者在当前架构下名不副实，反而误导（暗示存在可信的 sub-agent 实例可数）。
> 2. 优先级 **高→中**：主要场景下退化为自检，价值打折；待真实多 sub-agent 运行时再提级。
> 3. 实现说明里加一句边界声明：本维度在单 agent 串行模式下退化为"agent 复核自己 output 条数"，防的是 claim 笔误/漂移，不防系统性谎报（后者由 X2 盲区声明覆盖）。

### S2 · Plan 到 Execute 之间缺"预研"环节

**评审裁决：⏸️ 暂不建「阶段」，降为 scheduler-rules 一段调度指引**

**现状**：生命周期 Stage 2 Plan 产出 phase-plan.md 后直接进入 Stage 3 Execute（make packet → dispatch）。没有判断"这个 phase 是否需要先派人侦察"的机制。Orient 读取的是已有 Registry 风险段，不是对新任务的领域预研。

**wiki 依据**：[[预研Sub-agent委托]]（踩坑 #16："每新门类 dispatch 前必须预研"——文学门类原规划 3 agent ~100 条，预研后发现实际 ~180 条，切分偏差导致返工）。[[三阶段编排模式]] 阶段 0 即"预处理（分组+搜索词包）"。

**原建议**：在 Plan 与 Execute 之间加可选子阶段 **Pre-flight（预飞）**。Plan 阶段在 phase-plan.md 末尾声明一条判断：`pre_flight_needed: true/false`……（详见初稿）

**改动范围**：`workflow-steps.md` Stage 2 出口 + Stage 3 入口之间加 Pre-flight 段 + `templates/phase-plan.md` 加 `pre_flight_needed` 字段 + `glossary.md` 加 Pre-flight 术语。

> **【评审裁决】** 问题是真的（踩坑 #16 的切分偏差返工是实证），但**方案过重**：新增子阶段 + `pre_flight_needed` 字段 + glossary 术语，对当前成熟度的 skill 是 over-engineering。
>
> 更轻的替代：Plan 遇到不熟领域时，**第一个 task 就是侦察 task**（用现有 task-packet，mode=emergent，Allowed Tools=Read/Grep/Web）。这不需要新阶段，只需 `scheduler-rules.md` 加一段调度指引（"新 domain / 不熟代码库时，首 task 应为 recon，其 output 喂回 Plan 修正切分"），吃下约 80% 的价值且零结构改动。
>
> **裁决**：本轮**不建 Pre-flight 阶段、不加字段、不加术语**。降为 `scheduler-rules.md` 一段指引。若未来真实并行规模到来再升级为正式子阶段。

### S3 · Phase-plan 不支持同 Phase 内异质任务分组

**评审裁决：❌ 不做，归入 Non-Goals**

**现状**：Phase-plan 对整个 phase 声明统一的 strategy（serial / parallel / hybrid），所有 tasks 一视同仁。

**wiki 依据**：[[三阶段编排模式]]——异质任务批次需按任务类型/难度分组。[[上下文工程]]——不同任务类型消耗的 attention budget 不同。

**原建议**：Phase-plan 模板允许在 strategy 段声明 **task groups**，每个 group 有独立的 `strategy`、`group_size`……（详见初稿）

> **【评审裁决】** group 级 strategy + 跨组串行化是**调度器级功能**，而 enloom 的 Non-Goals 明确写了"不做 scheduler / CLI / 常驻 agent"。更关键的是——**在单 agent 串行执行下，group 级并行/串行策略根本没有执行意义**（`scheduler-rules.md:53`：执行实际串行），只增加 phase-plan 模板面积和 Plan 阶段的规划负担。这是为 skill 尚不能有效运行的规模做的前瞻设计。
>
> **裁决**：明确归入 Non-Goals（见 §5）。当真实多 sub-agent 并行运行成为常态后，再据实评估。

### S4 · health-check 每 Stage 转移都全量跑，attention budget 开销大

**评审裁决：✅ 采纳，优先级 中（置信度最高、改动最小、零风险）**

**现状**：v0.4 把 health-check 升级为"Stage 转移硬闸门执行器"——每次 1→2→3→4→5→6 边界都跑 health-check（`workflow-steps.md:245` "run at every stage boundary"）。全量九项检查（`workflow-steps.md:250-259`）。

**wiki 依据**：[[上下文工程]]——"Attention Budget 是有限资源。'Just in time' 优于全量前置。"

**建议**：将 health-check 拆为两档：
- **轻量档（Stage 转移时）**：只跑文件存在性闸门——`ls runs/<TASK>/task.md` 之类的机械检查，一行确认即可。v0.4 的硬闸门不变，但执行成本降到最低。
- **完整档（Orient 起 + 定期 drift）**：全量九项检查，保留在 Stage 1 Orient 和 Stage 4 Verify 周期触发。

轻量档通过时输出单行确认（"Gates OK: Stage 2→3, files present"），不再展开全量报告。

**改动范围**：`workflow-steps.md` Health Check 段（拆两档 + 触发条件）+ `landing-contract.md` §4（health-check 作为硬闸门的执行方式改为轻量档）。

> **【评审裁决】** 最不言自明的优化——Stage 转移时 95% 的情况只需"闸门文件在不在"，却展开九项全量报告。拆两档不改变 v0.4 语义（硬闸门仍在），只降执行成本。**改动小、零回归风险、收益直接**。维持优先级「中」，但它是本轮置信度最高的一项。

### S5 · Compaction 触发是可选检查，非必执行闸门

**评审裁决：✅ 采纳，优先级 高（最干净的采纳项）**

**现状**：Registry-and-compaction 定义"compaction 在 Stage 5 Integrate 后检查 trigger，如超标则建议 compaction"（`registry-and-compaction.md:136-139` "If met, run compaction"——可选措辞）。`workflow-steps.md:190` Stage 5 出口闸门只要求 "compaction trigger check run"（只检查不执行）。实践中 compaction 被推迟——项目状态持续膨胀。

**wiki 依据**：[[上下文工程]]——"如果不控制，上下文膨胀会从聊天窗口转移到文档系统。" [[落盘时序契约]]——"每 Stage 出口闸门必须是机械检查。"

**建议**：将 Compaction 升级为 Integrate 出口闸门条件之一（与 project_state/Registry 更新并列）。规则：
- Integrate 完成后，检查 project_state 是否超 200 行或 `## Accepted Results` 超 10 条
- 超标 → Compaction 必执行，未执行不准 Stage 5 出口（健康检查轻量档验证 `archive/phase-history.md` 的时间戳 ≤ 上次 Integrate 之后）
- 未超标 → 跳过（记录 "compaction not triggered" 即可）

**改动范围**：`registry-and-compaction.md`（触发从"建议"改"必执行"）+ `workflow-steps.md` Stage 5 出口闸门（加 compaction check）+ `landing-contract.md` §1 闸门表 Stage 5（改 "compaction trigger check run" → "compaction run or threshold not met"）。

> **【评审裁决】** 这是 enloom 自己 v0.3→v0.4「铁律→机械闸门」转换模式（Law 2/5 已先例）的又一个实例，与运行规模无关。可选 compaction 会让 Registry 无限膨胀，"上下文膨胀从聊天窗口转移到文档系统"。**最干净的采纳项**。
>
> 唯一提醒：触发阈值（200 行 / 10 条）写死可以，但在 `registry-and-compaction.md` 留一句"阈值是启发式、非教条"，避免变成不可讨论的硬数字。

### S6 · Promise Registry 的"存储是否容忍悬空引用"判断缺决策模板

**评审裁决：✅ 采纳，优先级 低（低成本脚手架，无害）**

**现状**：Promise Registry 机制设计完整——forward-declare output + tolerate dangling references + verify at end（`registry-and-compaction.md:95-101` 有退化机制）。但 Plan 阶段的决策"does this task's reference layer tolerate dangling references?"（`registry-and-compaction.md:102`）完全依赖 agent 经验判断，无任何模板化指引。`phase-plan.md:33-42` Promise Registry Draft 段只有一行问题，无结构化决策表。

**wiki 依据**：[[承诺页面机制]]——同一机制区分了"临时死链"（承诺页面）和"真死链"（需修复）。

**建议**：Phase-plan 模板新增 **引用容忍度决策表**，放在 Promise Registry draft 之前。至少覆盖 3-4 种常见引用类型作为示例（Obsidian wikilinks / 代码 import / 文件系统路径 / 数据库外键 / JSON schema $ref），agent 按项目类比填写。填完后再决定 Promise 是否可用。

**改动范围**：`templates/phase-plan.md`（加引用容忍度决策表）+ `registry-and-compaction.md` §3（引用此表）。

> **【评审裁决】** 3-4 行示例表是低成本脚手架，省下 agent 每次重新推导。典型的"有用的辅助"，无害。维持「低」。

### S7 · Task packet 的约束密度与任务复杂度不匹配

**评审裁决：✅ 采纳，优先级 低（把已有的 mode 概念推到逻辑终点）**

**现状**：Task packet 模板对所有任务提供同一套结构。`task-packet.md:4` 已定义三种 mode（emergent / recorded / audited），`workflow-steps.md:128-133` 有 selection 指引，但**约束密度对所有模式相同**——`workflow-steps.md:150` 对所有模式（含 emergent 探索）都要求 Forbidden 显式枚举 serial-integration 文件，对探索任务是过度工程。

**wiki 依据**：[[任务包设计模式]]——约束密度与 sub-agent 成功率的关系。[[路由预填模式]]——不同任务类型的约束粒度应不同。

**建议**：Task packet 模板按三种 mode 差异化字段要求：

| 字段 | emergent | recorded | audited |
|------|----------|----------|---------|
| Goal / Anti-Goal / Input / Output / Done Signal | 必填 | 必填 | 必填 |
| Writable | 必填（可简） | 必填（完整） | 必填（完整+检查方法） |
| Forbidden | 可选 | 必填 | 必填（显式枚举 serial-integration） |
| Acceptance | 必填（可简） | 必填 | 必填 |
| Review Budget | 可选 | 必填 | 必填 |
| Required Verification | 可选 | 可选 | **必填**（含 check_item 列表） |
| Independent Verification | 可选 | 建议 | **必填**（align S1） |

emergent 模式不填 Forbidden/Required Verification 仍合法；audited 模式缺 Required Verification 则 make-prompt 自检不通过。

**改动范围**：`templates/task-packet.md`（三模式差异化字段表）+ `scheduler-rules.md`（mode 选择指引更新）+ `workflow-steps.md` Stage 3（make-prompt 自检规则）。

> **【评审裁决】** 把已有的 mode 概念推到逻辑终点——emergent 不需要详细 Forbidden，audited 必须有 Required Verification，这是 mode 概念本应有的样子。改动适中，值得做。维持「低」。
>
> 注：表中的 "Independent Verification 必填（audited）" 依赖 S1 落地。若 S1 按裁决改为"report-vs-output 一致性校验"，此处字段名同步改为 "Claim Consistency Check"，但必填性不变。

## 4. 交叉局限（与 clear-mind 共享）

以下两条同时影响 enloom 和 clear-mind，来源相同，在两个 skill 的优化文档中同时记录。

### X1 · 交接方向单向，缺回退路径

**评审裁决：❌ 不做正式协议，最多 Failure Protocol 加一行人话建议**

**现状**：enloom triage 可建议先跑 clear-mind；clear-mind verdict=WORTH-IT 可建议进 enloom。但 enloom lifecycle 内部 phase 失败后，没有"降级回 clear-mind 重判方案可行性"的机制。当前 enloom Failure Protocol（`workflow-steps.md:225-239`）只处理 worker 级失败（repair packet / narrow task / rewrite goal / assumption review），不处理"这个 phase 的方向本身可能错了"的宏观降级。

**wiki 依据**：[[编组工具交接模式]]——"skill 的'终态'是默认值，不是铁律。"

**原建议**：enloom 侧 Failure Protocol 加一条 macro 级：Phase failed three times → write "direction-review-needed" finding into Registry `## Pending Dependencies`，建议退回 clear-mind。clear-mind 侧 review.md Handoff 段加"接收 enloom 退回"场景。

**改动范围**（enloom 侧）：`workflow-steps.md` Failure Protocol + `registry-and-compaction.md`。

> **【评审裁决】** 这是**投机的、双向耦合的、低触发频率**的功能。要真正工作需要双边协同（enloom 写 marker + clear-mind 读 marker），而 clear-mind 尚未全局安装、两者从未联合跑过。两个 skill 当前都明确"正交、可独立运行"，引入正式交接 marker 契约是**过早集成**。
>
> 现有 "3 次失败 → assumption review"（`workflow-steps.md:239`）已表达了"停下来反思方向"的意图。
>
> **裁决**：**不建协议、不定义 marker 格式、不动 Registry 语义**。最多在 Failure Protocol 加一句人话（"若怀疑方向本身错了，可考虑回 clear-mind 重判"），作为 agent 的可选提示，不构成机械闸门。归入 Non-Goals。

### X2 · 单 agent 隔离 = 纯靠文件约束的诚实盲区未声明

**评审裁决：✅ 采纳，优先级 高（性价比最高），且补一个文档未提的实证盲区**

**现状**：enloom Evidence Contract §Honest Blind Spot（`evidence-contract.md:76-78`）已声明一项：*"worker isolation is enforced by packet field discipline, not by execution isolation — there is no independent subagent runtime."* 但声明只提了"不能真实验证 worker 是否碰了 forbidden files"。未覆盖更广泛的不可靠性。worker-report 模板（`worker-report.md:27-29`）的 Known Blind Spots 段呼应了这一项，但无独立 `honest-limitations.md` 文件。

clear-mind **完全没有平行声明**。

**wiki 依据**：[[Sub-agent计数不可信]]（方向不固定的系统性偏差）+ [[上下文工程]]（信任度从 40% 降至 29%）。

**建议**：Evidence Contract Known Blind Spots 模板扩展。

**改动范围**（enloom 侧）：`evidence-contract.md` §Honest Blind Spot + `templates/worker-report.md`。

> **【评审裁决】** 性价比最高的一项——两行改动换来可信度的实质性提升。
>
> **评审还发现一个文档未提、但更具体的盲区**：enloom 的并行 dispatch 在单 agent 下是**虚拟/串行的**——`scheduler-rules.md:53` 已明确承认"无法真正并行 dispatch 多个写 worker……执行实际串行"，但这个诚实声明**没有进 Honest Blind Spot**。也就是说：当前 phase-plan 里声明的 `strategy: parallel` + Ownership Table 在单 agent 下只是协议形式，没有任何运行时并行被真正执行。这比文档原建议的 "cross-role verification" 更可操作、更值得写进去。
>
> **裁决**：Known Blind Spots 扩展为**三项**（而非文档原建议的两项）：
> 1. "cross-worker real isolation — single-agent environment, no independent runtime"（已有）
> 2. "cross-role verification — verdict/review/audit may all come from the same model context; independent reasoning-chain verification is not guaranteed"（文档原意）
> 3. **"virtual parallelism — in single-agent mode, declared `strategy: parallel` is protocol form only; execution is actually serial (`scheduler-rules.md:53`)"（评审新增，实证依据）**
>
> 第 3 项应同时回写到 `scheduler-rules.md` 该行附近，使其从"埋在规则里的一句话"升级为"明示的架构盲区"。
